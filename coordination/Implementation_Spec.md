# ZenPulse Implementation Specification

## 1. Status And Purpose

Этот документ является координационной надстройкой над исходной документацией проекта.

Он:
- не заменяет документы из `docs/`;
- не имеет права противоречить `docs/`;
- нужен для реализации, декомпозиции задач, контроля полноты и предотвращения дрейфа требований;
- должен отправлять исполнителя обратно в конкретные документы `docs/` перед любым кодом.

Единственный источник истины по продукту и реализации:
- `docs/Product.md`
- `docs/UX.md`
- `docs/UI.md`
- `docs/Screens.md`
- `docs/State_and_Navigation.md`
- `docs/Stack.md`
- `docs/AI_Content.md`

Если текущая спецификация конфликтует с любым из файлов в `docs/`, приоритет всегда у `docs/`.

## 2. Mandatory Workflow Before Any Code Change

Перед любой задачей реализации нужно проходить одну и ту же последовательность:

1. Открыть `coordination/Live_Plan.md` и найти текущий активный этап.
2. Открыть этот файл и определить рамку текущего этапа.
3. По ссылкам из этого файла перейти в конкретные документы `docs/`.
4. Реализовывать только то, что разрешено текущим этапом и подтверждено исходной документацией.
5. После завершения задачи обновить `coordination/Live_Plan.md`.

Запрещено:
- писать код по памяти без возврата к `docs/`;
- расширять scope сверх MVP;
- добавлять новые продуктовые сущности без прямого указания в `docs/`;
- трактовать эту спецификацию как новую source of truth.

## 3. Project Frame

### 3.1. Product Type

ZenPulse реализуется как качественный интерактивный MVP-прототип мобильного Android-приложения в категории digital wellness / meditation.

Это не production-версия.

### 3.2. Core Value

Проект должен продемонстрировать:
- аккуратную мобильную вёрстку;
- цельный premium visual style;
- понятную модель Premium-доступа;
- реалистичную AI/vibe-фичу без реальной AI-интеграции;
- предсказуемую навигацию;
- сохранение симулированной подписки между запусками.

### 3.3. Fixed Platform

Фиксированные условия:
- платформа: Android;
- основной сценарий проверки: Android Studio emulator, Pixel 5, Android 13;
- язык интерфейса: русский;
- тема: тёмная;
- проект должен ощущаться как мобильное приложение, а не как адаптированный web UI.

## 4. Non-Negotiable Product Boundaries

### 4.1. What Must Exist

В приложении обязательно должны быть:
- ровно 3 основных экрана;
- нижняя tab-навигация;
- каталог из 6 карточек медитаций;
- разделение на free и premium контент;
- Paywall с двумя тарифами;
- симулированная локальная активация подписки;
- AI Mood как отдельный premium-экран;
- сохранение состояния подписки после перезапуска;
- единый soft wellness визуальный язык;
- общий фон приложения;
- 6 отдельных изображений для карточек медитаций.

### 4.2. What Must Not Exist

Запрещено добавлять:
- backend;
- реальный AI API;
- реальный billing/store flow;
- авторизацию;
- профиль;
- onboarding;
- аудиоплеер;
- detail screens;
- чаты;
- историю AI-генераций;
- дополнительные пользовательские экраны сверх трёх утверждённых;
- сложную архитектуру ради будущего масштабирования.

## 7. Core User Flows

### 7.1. Primary Premium Unlock Flow

Базовый путь:
1. Пользователь открывает приложение.
2. Попадает на `MeditationsScreen`.
3. Видит 2 free и 4 locked карточки.
4. Нажимает на locked premium-карточку.
5. Попадает на `PaywallScreen`.
6. Видит Premium benefits и 2 тарифа.
7. Нажимает CTA.
8. Подписка активируется локально.
9. Пользователь возвращается в понятное состояние.
10. Premium-карточки становятся визуально доступными.

Предпочтительный возврат после активации:
- программный переход на `MeditationsScreen`.

### 7.2. Premium AI Flow

Путь без подписки:
1. Пользователь нажимает таб `Настрой дня`.
2. `AIMoodScreen` не должен открыться даже временно.
3. Пользователь переводится на `PaywallScreen`.

Путь с подпиской:
1. Пользователь нажимает таб `Настрой дня`.
2. Открывается `AIMoodScreen`.
3. Пользователь выбирает настроение.
4. Нажимает кнопку генерации.
5. Видит короткий loading.
6. Получает результат в отдельной карточке.

### 7.3. Premium Status Flow

После активации подписки:
- таб `Premium` остаётся видимым;
- он продолжает вести на `PaywallScreen`;
- экран не должен выглядеть как ещё один блокирующий paywall;
- экран должен показывать, что доступ уже активен.

## 8. Screen Contracts

### 8.1. MeditationsScreen Contract

Назначение:
- главный экран приложения;
- каталог медитаций;
- первая точка входа;
- ключевая точка демонстрации premium-логики.

Обязательная структура:
1. safe area top spacing
2. кастомный заголовок
3. краткий подзаголовок
4. компактный блок статуса доступа
5. вертикальный список из 6 карточек
6. нижний safe padding

Обязательные данные:
- 6 карточек;
- 2 free;
- 4 premium;
- у каждой карточки: изображение, название, длительность.

Обязательное поведение:
- free-карточки остаются доступными всегда;
- free-карточки не создают новый экран;
- premium-карточки без подписки выглядят locked и ведут на Paywall;
- premium-карточки с подпиской выглядят как обычные доступные карточки;
- locked-карточки не выглядят как мёртвые disabled элементы.

Что нельзя делать:
- detail view медитаций;
- фильтры;
- категории;
- плеер;
- дополнительные действия внутри карточек;
- иной основной layout, кроме вертикального списка.

Основные первоисточники:
- `docs/Product.md`
- `docs/UX.md`
- `docs/UI.md`
- `docs/Screens.md`
- `docs/State_and_Navigation.md`

### 8.2. PaywallScreen Contract

Назначение:
- продажа ценности Premium;
- симуляция активации подписки;
- статусный экран Premium после активации.

Обязательная структура:
1. safe area top spacing
2. hero-зона
3. подзаголовок или продающее описание
4. блок преимуществ
5. блок тарифов
6. основной CTA-блок
7. короткая secondary подпись
8. нижний safe padding

Обязательные данные:
- 2 тарифа: месячный и годовой;
- по умолчанию выбран годовой;
- годовой тариф выделен как выгодный;
- 3 или 4 преимущества Premium.

Режим без подписки:
- CTA активен;
- можно выбрать тариф;
- можно локально активировать подписку;
- после нажатия состояние должно явно измениться.

Режим с подпиской:
- экран показывает активный доступ;
- CTA не запускает повторную покупку;
- тариф может оставаться декоративно выбранным;
- экран сохраняет ценность, но убирает ощущение блокировки.

Что нельзя делать:
- real billing;
- restore purchase;
- сложный юридический store-like flow;
- отдельный экран успешной покупки;
- отдельный экран ошибки покупки;
- недоступный CTA на маленьких экранах.

Основные первоисточники:
- `docs/Product.md`
- `docs/UX.md`
- `docs/UI.md`
- `docs/Screens.md`
- `docs/State_and_Navigation.md`

### 8.3. AIMoodScreen Contract

Назначение:
- компактная premium AI/vibe-feature;
- ощущение персонализации;
- эмоциональная ценность подписки.

Условие доступа:
- экран полноценного взаимодействия допустим только при `isSubscribed = true`.

Обязательная структура:
1. safe area top spacing
2. кастомный заголовок
3. краткое объяснение функции
4. блок выбора настроения
5. основная кнопка генерации
6. блок результата
7. нижний safe padding

Обязательные входы:
- 3 настроения:
- `Спокойно`
- `Нужен фокус`
- `Тревожно`

Обязательное поведение:
- до выбора настроения генерация недоступна;
- после выбора ровно одно настроение активно;
- при запуске генерации появляется loading;
- текст выбирается только из пула выбранного настроения;
- результат рендерится на том же экране;
- повторная генерация допустима без усложнения сценария.

Что нельзя делать:
- чат;
- ручной ввод;
- история;
- streaming;
- отдельные AI sub-screens;
- длинные тексты;
- real AI API.

Основные первоисточники:
- `docs/Product.md`
- `docs/UX.md`
- `docs/UI.md`
- `docs/Screens.md`
- `docs/State_and_Navigation.md`
- `docs/AI_Content.md`

## 9. State Specification

### 9.1. Global State

Глобальными должны быть только:
- `isSubscribed: boolean`
- `isSubscriptionHydrated: boolean`

Смысл:
- `isSubscribed` — единый источник правды для premium-доступа;
- `isSubscriptionHydrated` — защита от неверной отрисовки доступа до завершения чтения storage.

### 9.2. Local Paywall State

Локально на Paywall:
- `selectedPlan: 'monthly' | 'yearly'`
- pressed/select UI state

Стартовое значение:
- `selectedPlan = 'yearly'`

### 9.3. Local AI State

Локально на AI Mood:
- `selectedMood: 'calm' | 'focus' | 'anxious' | null`
- `generatedAffirmation: string | null`
- `isGenerating: boolean`

### 9.4. Meditations Local State

Допустимо только локальное UI state:
- pressed states
- animation feedback
- временные визуальные эффекты

Запрещено:
- отдельное data state для каталога;
- фильтрация;
- дополнительные access flags.

## 10. Single Source Of Truth Rule

Вся access-логика должна строиться только вокруг `isSubscribed`.

`isSubscribed` влияет на:
- locked-статус premium-карточек;
- возможность открыть `AIMoodScreen`;
- поведение вкладки `Настрой дня`;
- режим `PaywallScreen`;
- доступность AI-генерации.

Запрещено создавать дополнительные параллельные флаги доступа:
- `canOpenAIMood`
- `hasPremiumMeditationsAccess`
- любые другие дублирующие access booleans

## 11. Persistence Contract

### 11.1. Persisted Data

Между перезапусками сохраняется только:
- `isSubscribed`

### 11.2. Storage Technology

Использовать:
- `AsyncStorage`

### 11.3. Storage Isolation

Чтение и запись storage должны быть вынесены в отдельный модуль.

Нельзя:
- обращаться к `AsyncStorage` хаотично из нескольких экранов;
- смешивать storage и presentation в одном месте без причины.

### 11.4. Hydration Rule

До завершения гидратации:
- нельзя показывать логически неправильное состояние доступа;
- нельзя допускать flicker locked/unlocked;
- нельзя временно открывать AI Mood до проверки подписки.

Допустимое решение:
- короткое нейтральное loading state;
- либо скрытие чувствительных к подписке частей до hydration completion.

## 12. Navigation Contract

### 12.1. Allowed Transitions

Разрешённые переходы:
- `MeditationsScreen` -> `PaywallScreen` по locked premium-card tap
- `MeditationsScreen` -> `PaywallScreen` по tab `Premium`
- `MeditationsScreen` -> `AIMoodScreen` по tab `Настрой дня`, если есть подписка
- `MeditationsScreen` -> `PaywallScreen` по tab `Настрой дня`, если подписки нет
- `PaywallScreen` -> `MeditationsScreen` по tab `Медитации`
- `PaywallScreen` -> `AIMoodScreen` по tab `Настрой дня`, если подписка активна
- `AIMoodScreen` -> `MeditationsScreen` по tab `Медитации`
- `AIMoodScreen` -> `PaywallScreen` по tab `Premium`

### 12.2. Preferred Gate For AI Mood

Предпочтительный способ:
- перехватывать нажатие на вкладку `Настрой дня`;
- если подписки нет, отменять обычный переход;
- переводить пользователя на `Premium` / `PaywallScreen`.

Нельзя:
- сначала открывать `AIMoodScreen`, а потом редиректить;
- показывать полурабочий AI screen без доступа.

### 12.3. Post-Activation Navigation

После `activateSubscription()`:
- `isSubscribed` должен стать `true`;
- состояние сохраняется;
- locked premium-контент должен сразу обновиться;
- AI Mood должен стать доступен без рестарта;
- предпочтителен программный переход на `MeditationsScreen`.

## 13. Data Contract

### 13.1. Meditations Data

Нужен статический локальный список из 6 объектов.

Для каждой карточки обязательно:
- `id`
- `title`
- `duration`
- `isPremium`
- `image`

Ожидаемое распределение:
- 2 карточки `isPremium = false`
- 4 карточки `isPremium = true`

### 13.2. Plans Data

Нужен локальный набор из 2 тарифов:
- monthly
- yearly

Каждый тариф должен иметь:
- ключ;
- русское название;
- цену;
- короткую поясняющую подпись;
- опциональный признак выгодности для yearly.

### 13.3. AI Mood Data

Нужны:
- список mood options;
- объект с массивами ответов по mood key.

Обязательные mood keys:
- `calm`
- `focus`
- `anxious`

Каждому mood key должен соответствовать свой набор подготовленных текстов.

## 14. AI Content Contract

### 14.1. Mood Mapping

Обязательная смысловая карта:
- `calm` -> `Спокойно`
- `focus` -> `Нужен фокус`
- `anxious` -> `Тревожно`

### 14.2. Response Selection Rule

Логика генерации:
1. определяется выбранный mood;
2. берётся соответствующий массив;
3. выбирается один текст из этого массива;
4. текст из другой категории использовать нельзя.

### 14.3. Response Tone

Каждый AI-текст должен быть:
- на русском языке;
- мягким;
- спокойным;
- поддерживающим;
- неброским;
- правдоподобным;
- wellness-oriented.

Текст не должен быть:
- агрессивным;
- слишком мотивационным;
- эзотерическим;
- медицинским;
- назидательным;
- техническим;
- машинно-шаблонным.

### 14.4. Response Length

Целевое ограничение:
- 1-3 коротких предложения;
- желательно до 180 символов;
- абсолютный максимум до 240 символов только если UI не ломается.

### 14.5. Loading Delay

Имитация генерации должна включать искусственную задержку:
- ориентир 700-1200 мс

### 14.6. AI Fallback

Полноценный error state не нужен.

Допускается только безопасный резервный текст, если по какой-то причине корректный ответ не выбран.

## 16. Interaction Contract

### 16.1. Pressed Feedback

Обязателен для:
- кнопок;
- карточек медитаций;
- mood options;
- plan cards;
- tab items, если визуально уместно.

### 16.2. Selected States

Должны быть явно различимы для:
- активного тарифа;
- выбранного настроения;
- активной вкладки.

### 16.3. Locked States

Locked-состояние должно:
- быть очевидным;
- оставаться эстетичным;
- не превращать элемент в серую недоступную массу;
- сохранять ощущение tappable-пути к Premium.

### 16.4. Loading States

Loading нужен только там, где это обосновано:
- AI Mood generation
- краткая гидратация подписки при старте, если это нужно для корректного UX

## 17. Mobile Layout Rules

### 17.1. Safe Area

Каждый экран обязан:
- уважать верхнюю и нижнюю safe area;
- не конфликтовать с системными зонами;
- не прижимать контент к краям.

### 17.2. Scroll Behavior

Ключевые экраны должны устойчиво скроллиться.

Особенно:
- Paywall;
- AI Mood при длинном результате;
- Meditations при ограниченной высоте.

### 17.3. Absolute Layout Restriction

Запрещено строить основной layout на `position: absolute`.

Допускается только:
- некритичный декоративный слой;
- overlay;
- неструктурный визуальный акцент.

### 17.4. Fixed Height Restriction

Нельзя опираться на layout, работающий только при одной высоте экрана.

### 17.5. Touch Targets

Интерактивные элементы должны быть комфортными для пальца.

Минимальный ориентир:
- около 44-48 px
