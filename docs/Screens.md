# Screens.md

## 1. Назначение документа

Этот документ фиксирует точную структуру и поведение каждого экрана ZenPulse.

Цель документа:
- дать однозначное описание всех 3 экранов;
- исключить произвольные трактовки layout и UX;
- зафиксировать, какие блоки есть на каждом экране;
- зафиксировать, какие состояния есть у каждого экрана;
- зафиксировать все ключевые действия пользователя и результат этих действий.

Документ синхронизирован с:
- `Product.md`
- `Stack.md`
- `UX.md`
- `UI.md`

При конфликте между декоративностью и устойчивостью мобильного layout необходимо выбирать устойчивость layout.

---

## 2. Общие правила для всех экранов

### 2.1. Общий набор экранов
В приложении существуют **ровно 3 основных экрана**:
1. `MeditationsScreen`
2. `AIMoodScreen`
3. `PaywallScreen`

Дополнительные продуктовые экраны, скрытые подэкраны, отдельные detail-экраны карточек и вторичные flow не создаются.

### 2.2. Общие правила layout
Для всех экранов обязательно:
- использовать safe area;
- использовать единый фон приложения;
- использовать затемняющий или смягчающий overlay поверх фона;
- использовать мобильные внутренние отступы;
- не строить основной layout на `absolute`;
- не использовать жёсткие высоты секций без необходимости;
- сохранять корректную читаемость поверх фонового изображения.

### 2.3. Общие правила заголовков
На всех экранах используются **кастомные заголовки внутри контента**, а не стандартные системные header-бары React Navigation.

Общие требования к заголовкам:
- заголовок крупный и заметный;
- подзаголовок краткий;
- текст полностью на русском языке;
- заголовки должны работать на ограниченной ширине экрана.

### 2.4. Общие правила прокрутки
Сценарий прокрутки должен быть предсказуемым.

Обязательно:
- если контент не помещается по высоте, экран должен корректно скроллиться;
- Paywall должен быть построен с расчётом на вертикальный scroll;
- на экранах нельзя допускать недоступность CTA из-за ограниченной высоты.

### 2.5. Общие правила интерактивности
Все интерактивные элементы должны:
- иметь достаточный touch target;
- иметь заметное pressed-состояние;
- не быть декоративными псевдо-кнопками;
- быть однозначно распознаваемыми как нажимаемые элементы.

### 2.6. Общие правила анимаций
Допускаются мягкие микроанимации:
- нажатие карточек;
- выбор тарифа;
- выбор настроения;
- появление результата AI Mood.

Анимации должны быть короткими, деликатными и не мешать стабильности интерфейса.

## 3. Навигационная модель экранов

### 3.1. Нижняя навигация
Приложение использует **bottom tabs**.

Вкладки:
- `Медитации`
- `Настрой дня`
- `Premium`

### 3.2. Соответствие вкладок и экранов
- вкладка `Медитации` открывает `MeditationsScreen`;
- вкладка `Настрой дня` ведёт к `AIMoodScreen`, но при отсутствии подписки перенаправляет пользователя на `PaywallScreen`;
- вкладка `Premium` всегда открывает `PaywallScreen`.

### 3.3. Стартовый экран
При первом запуске приложение открывается на `MeditationsScreen`.

### 3.4. Поведение после активации подписки
После активации подписки:
- premium-медитации становятся доступными;
- вкладка `Настрой дня` открывает `AIMoodScreen` без блокировки;
- вкладка `Premium` по-прежнему остаётся доступной и ведёт на `PaywallScreen`, но экран должен уже показывать состояние активного доступа, а не состояние блокировки.

## 4. Экран Meditations

### 4.1. Роль экрана
`MeditationsScreen` — это главный экран приложения.

Он должен:
- быть первой точкой входа в приложение;
- показывать каталог контента;
- объяснять, что часть медитаций доступна бесплатно, а часть относится к Premium;
- быть источником перехода на Paywall через locked-контент.

### 4.2. Основная задача экрана
Экран должен быстро и понятно отвечать на вопросы пользователя:
- что это за приложение;
- что здесь можно открыть прямо сейчас;
- что относится к premium;
- как перейти к платному доступу.

### 4.3. Общая структура экрана сверху вниз
Экран состоит из следующих блоков сверху вниз:
1. Safe area отступ
2. Кастомный заголовок
3. Короткий вступительный текст / подзаголовок
4. Небольшой информационный блок о Premium-доступе
5. Список карточек медитаций
6. Нижний safe padding под bottom tabs

### 4.4. Кастомный заголовок
Заголовочная зона должна содержать:
- основной заголовок;
- короткий подзаголовок;
- опционально небольшой визуальный акцент или бейдж категории.

Рекомендуемая смысловая подача:
- заголовок про медитации и внутренний баланс;
- подзаголовок о выборе практики на текущий момент.

Заголовок не должен быть слишком длинным и не должен занимать чрезмерно много вертикального пространства.

### 4.5. Информационный блок о доступе
Под заголовком должен быть компактный surface-блок, который объясняет состояние доступа.

Если подписка не активна:
- блок сообщает, что часть медитаций доступна только в Premium;
- блок может содержать мягкий премиальный акцент;
- блок не должен выглядеть как ошибка или предупреждение.

Если подписка активна:
- блок сообщает, что Premium активен;
- блок может использовать success/premium-стилистику;
- состояние должно быть визуально позитивным.

### 4.6. Список карточек
На экране отображается **6 карточек медитаций**.

Распределение:
- 2 карточки free;
- 4 карточки premium.

Карточки должны быть расположены в одном вертикальном списке.

Допускается только один основной вариант раскладки:
- вертикальный список карточек, оптимизированный под мобильный scroll.

Не использовать:
- сложные masonry-сетки;
- двухколоночную сетку с риском ухудшения читаемости;
- горизонтальные карусели как основной способ показа контента.

### 4.7. Содержимое карточки медитации
Каждая карточка должна содержать:
- фоновое изображение карточки;
- overlay для читаемости;
- название медитации;
- длительность;
- при необходимости дополнительный малый бейдж статуса.

Контент карточки должен быть компактным и хорошо читаться поверх изображения.

### 4.8. Поведение free-карточек
Для доступных карточек:
- карточка выглядит активной;
- не затемнена;
- имеет обычный pressed-state;
- нажатие не ведёт на Paywall.

В рамках MVP free-карточка не обязана открывать отдельный детальный экран.

Допустимое поведение free-карточки:
- лёгкий pressed feedback без перехода;
- или спокойный локальный отклик/заглушка без создания нового экрана.

Приоритет: не создавать лишний flow за пределами утверждённых 3 экранов.

### 4.9. Поведение premium-карточек без подписки
Если подписка не активирована:
- premium-карточка визуально затемнена;
- поверх изображения есть locked-overlay;
- присутствует иконка замка или premium-метка;
- карточка остаётся явно нажимаемой;
- по нажатию пользователь переходит на `PaywallScreen`.

Важно:
- locked-карточка не должна выглядеть как полностью отключённый элемент;
- пользователь должен понимать, что карточка доступна после Premium.

### 4.10. Поведение premium-карточек после активации подписки
Если подписка активирована:
- затемнение и замок исчезают;
- premium-карточка начинает выглядеть как обычная доступная карточка;
- карточки становятся частью общего списка без визуальной блокировки.

### 4.11. Состояния экрана Meditations
Экран должен поддерживать следующие состояния:
- стандартное состояние без подписки;
- состояние после активации подписки;
- pressed-state отдельных карточек;
- устойчивое состояние на маленьком экране при длинном списке.

### 4.12. Что нельзя делать на MeditationsScreen
Нельзя:
- создавать detail-экраны медитаций;
- добавлять сложные фильтры и категории;
- добавлять аудиоплеер;
- делать слишком длинные описания карточек;
- перегружать верхнюю часть экрана декоративными блоками.

## 5. Экран Paywall

### 5.1. Роль экрана
`PaywallScreen` — экран premium-подписки.

Он должен:
- объяснять ценность Premium;
- визуально быть самым продающим экраном приложения;
- оставаться мобильным и устойчивым на маленьком экране;
- переводить пользователя в состояние активной подписки.

### 5.2. Когда пользователь попадает на экран
Пользователь попадает на `PaywallScreen` в следующих случаях:
- через вкладку `Premium`;
- при нажатии на locked-карточку на `MeditationsScreen`;
- при попытке открыть вкладку `Настрой дня` без активной подписки.

### 5.3. Общая структура экрана сверху вниз
Экран состоит из следующих блоков сверху вниз:
1. Safe area отступ
2. Hero-зона с крупным заголовком
3. Короткий подзаголовок / продающее описание
4. Блок преимуществ Premium
5. Блок выбора тарифов
6. Основной CTA-блок
7. Дополнительная короткая поясняющая подпись
8. Нижний safe padding

Экран должен поддерживать vertical scroll.

### 5.4. Hero-зона
Hero-зона должна производить первое впечатление “дорогого wellness-продукта”.

Содержимое hero-зоны:
- крупный заголовок;
- мягкий премиальный подзаголовок;
- при необходимости декоративный glow/gradient accent;
- без тяжёлой иллюстративной перегрузки.

### 5.5. Блок преимуществ Premium
Блок преимуществ должен быть компактным и легко сканируемым.

Рекомендуемое количество пунктов:
- 3 или 4 преимущества.

Каждый пункт содержит:
- маленькую иконку или маркер;
- короткий заголовок или краткую формулировку;
- без длинных описательных абзацев.

### 5.6. Блок тарифов
На Paywall должны быть показаны **2 тарифа**:
- месячный;
- годовой.

Тарифы должны отображаться как две отдельные карточки выбора.

Каждая карточка тарифа должна содержать:
- название тарифа;
- цену;
- краткую поясняющую подпись;
- selected-state.

Для годового тарифа:
- он должен быть визуально выделен как более выгодный;
- может иметь бейдж вроде «выгоднее» или аналогичную формулировку;
- выделение должно быть заметным, но не агрессивным.

### 5.7. Выбор тарифа
Поведение выбора:
- в один момент времени выбран только один тариф;
- выбранный тариф визуально отличается border/gradient/highlight-стилем;
- переключение между тарифами даёт мягкий pressed/select feedback.

Рекомендуемое стартовое состояние:
- по умолчанию выбран годовой тариф как выгодный.

### 5.8. Основной CTA
На экране должна быть одна основная CTA-кнопка.

Назначение кнопки:
- симулировать успешную активацию подписки.

Требования:
- CTA — самый заметный интерактивный элемент экрана;
- кнопка крупная и удобная для нажатия;
- визуально соответствует premium-стилистике;
- текст полностью на русском языке.

### 5.9. Поведение CTA без активной подписки
Если подписка ещё не активирована:
- нажатие на CTA активирует подписку локально;
- состояние подписки сохраняется;
- интерфейс должен отразить успешный результат;
- пользователь должен получить понятный переход обратно в премиум-доступ.

Предпочтительное поведение после успешной активации:
- показать краткое успешное состояние;
- затем вернуть пользователя к предыдущему пользовательскому пути или к контенту.

### 5.10. Поведение экрана после активации подписки
Если пользователь уже подписан и открывает вкладку `Premium`:
- экран остаётся доступным;
- он показывает состояние активного Premium;
- вместо ощущения блокировки должен быть статус уже открытого доступа.

Экран в активном состоянии может содержать:
- success/premium-блок о том, что доступ активен;
- сохранённый выбор тарифа как декоративное состояние;
- CTA в виде неактивного статуса или вторичной кнопки без повторного billing flow.

Важно:
- не делать повторную активацию обязательной;
- не делать confusing-состояние, в котором пользователь не понимает, что подписка уже действует.

### 5.11. Дополнительная подпись
Под CTA допускается короткая secondary-подпись.

Она должна:
- быть короткой;
- не напоминать юридический paywall store-экран;
- не создавать ощущения реального биллинга;
- быть частью мягкого demo/MVP-повествования.

### 5.12. Состояния экрана Paywall
Экран должен поддерживать следующие состояния:
- месячный тариф выбран;
- годовой тариф выбран;
- CTA доступна;
- подписка успешно активирована;
- активное Premium-состояние при повторном открытии после подписки.

### 5.13. Что нельзя делать на PaywallScreen
Нельзя:
- подключать реальный billing;
- добавлять restore purchase;
- добавлять юридически сложный store-flow;
- строить экран на критичных absolute-позициях;
- делать CTA недоступной на маленьком экране;
- перегружать экран большим количеством текста.

## 6. Экран AI Mood

### 6.1. Роль экрана
`AIMoodScreen` — отдельная premium-фича, создающая ощущение персонализации и AI-магии.

Экран должен:
- восприниматься как самостоятельный раздел приложения;
- быть простым и красивым;
- быстро объяснять, что пользователь может сделать;
- выдавать реалистичный мок-результат в стиле wellness.

### 6.2. Условие доступа
AI Mood является premium-функцией.

Поведение:
- без активной подписки вкладка `Настрой дня` не открывает этот экран, а перенаправляет пользователя на `PaywallScreen`;
- с активной подпиской вкладка `Настрой дня` открывает `AIMoodScreen` напрямую.

### 6.3. Общая структура экрана сверху вниз
Экран состоит из следующих блоков сверху вниз:
1. Safe area отступ
2. Кастомный заголовок
3. Короткое пояснение о функции
4. Блок выбора настроения
5. Кнопка генерации
6. Блок результата
7. Нижний safe padding

### 6.4. Заголовок и вводный текст
Верхняя часть экрана должна содержать:
- крупный заголовок;
- спокойный подзаголовок;
- объяснение, что пользователь может получить персональную короткую практику или аффирмацию.

Вводный текст должен быть кратким и не занимать слишком много вертикального пространства.

### 6.5. Блок выбора настроения
На экране должно быть **3 варианта настроения**.

Каждый вариант содержит:
- emoji;
- короткую подпись;
- заметное selected-состояние;
- удобную tappable-зону.

Настроения должны быть достаточно различимыми по эмоциональному смыслу.

### 6.6. Поведение выбора настроения
Поведение:
- в один момент времени выбран только один вариант;
- до выбора никакое настроение не считается активным;
- при выборе карточка или кнопка настроения получает визуальное выделение;
- повторный выбор другого настроения меняет активное состояние.

### 6.7. Кнопка генерации
На экране должна быть одна основная кнопка генерации.

Её задача:
- запустить выдачу реалистичного мок-ответа, привязанного к выбранному настроению.

Поведение:
- до выбора настроения кнопка не должна запускать генерацию;
- после выбора настроения кнопка становится готовой к действию;
- нажатие даёт понятный feedback.

### 6.8. Логика результата
Результат должен:
- зависеть от выбранного настроения;
- браться из заранее подготовленного пула мок-ответов;
- выглядеть как правдоподобный AI-output;
- быть коротким и хорошо помещаться в mobile UI.

Формат результата:
- короткая аффирмация;
- или короткий meditation-text;
- без длинных абзацев;
- без сложного форматирования.

### 6.9. Loading-состояние
После нажатия на кнопку генерации допускается короткое loading-состояние.

Оно нужно для того, чтобы:
- сделать поведение более правдоподобным;
- создать ощущение работы AI-функции;
- сделать появление результата более приятным.

Loading должен быть коротким и не раздражающим.

### 6.10. Блок результата
Результат выводится в отдельной surface-карточке.

Требования к карточке результата:
- хороший контраст;
- мягкий визуальный акцент;
- комфортный внутренний padding;
- текст не должен “ломать” layout;
- карточка должна адаптироваться к длине результата.

Если текст результата длиннее ожидаемого:
- экран должен сохранять стабильность;
- текст не должен обрезаться неконтролируемо;
- scroll экрана должен оставаться рабочим.

### 6.11. Состояния экрана AI Mood
Экран должен поддерживать следующие состояния:
- ничего не выбрано;
- выбрано одно настроение;
- идёт генерация;
- результат показан;
- повторная генерация после смены настроения.

### 6.12. Что нельзя делать на AIMoodScreen
Нельзя:
- превращать экран в полноценный AI-чат;
- добавлять историю сообщений;
- добавлять потоковую генерацию;
- добавлять текстовые поля ввода;
- делать слишком длинные ответы;
- использовать реальный AI API в рамках MVP.

## 7. Контент экранов

### 7.1. Язык интерфейса
Весь пользовательский интерфейс должен быть **полностью на русском языке**.

Это касается:
- заголовков;
- подзаголовков;
- кнопок;
- подписей;
- тарифов;
- преимуществ;
- текста AI Mood;
- системных microcopy-элементов.

### 7.2. Tone of voice
Тексты на всех экранах должны быть:
- спокойными;
- мягкими;
- современными;
- визуально компактными;
- без тяжёлой терапевтической риторики;
- без чрезмерной пафосности.

## 8. Mobile-специфичные требования по экранам

### 8.1. Safe area
На каждом экране контент должен корректно учитывать safe area.

Нельзя:
- прижимать заголовок к самому верху;
- накладывать контент на системные зоны;
- рассчитывать, что статус-бар не повлияет на layout.

### 8.2. Отсутствие критичного absolute layout
Нельзя строить ключевую структуру экранов через `absolute`.

Особенно важно для:
- Paywall;
- CTA-зоны;
- карточек и верхнего hero-блока.

### 8.3. Устойчивость к маленькому экрану
Все 3 экрана должны быть устойчивыми на ограниченной высоте.

Обязательно:
- ключевые элементы остаются доступными;
- CTA не исчезают за пределами экрана;
- текст не налезает друг на друга;
- интерфейс можно прокрутить, если контента становится больше.

### 8.4. Touch targets
Все карточки, тарифы, кнопки и mood-элементы должны быть удобны для нажатия пальцем.

Нельзя делать:
- мелкие кнопки;
- узкие зоны нажатия;
- едва заметные интерактивные элементы.

### 8.5. Длинный текст
Все экраны должны выдерживать slightly variable content length.

Особенно важно для:
- описаний на Paywall;
- названий карточек медитаций;
- результатов AI Mood.

Layout должен оставаться стабильным без визуальных поломок.

## 9. Итоговая матрица экранов

### 9.1. MeditationsScreen
Назначение:
- главный экран каталога.

Основные блоки:
- заголовок;
- подзаголовок;
- статус доступа;
- список из 6 карточек.

Ключевое поведение:
- free-карточки доступны;
- locked-карточки ведут на Paywall;
- после подписки locked-состояние исчезает.

### 9.2. PaywallScreen
Назначение:
- экран Premium и активации подписки.

Основные блоки:
- hero-зона;
- преимущества;
- 2 тарифа;
- CTA.

Ключевое поведение:
- выбор тарифа;
- активация подписки;
- после подписки экран показывает активный Premium-статус.

### 9.3. AIMoodScreen
Назначение:
- premium AI-фича.

Основные блоки:
- заголовок;
- описание;
- выбор одного из 3 настроений;
- кнопка генерации;
- карточка результата.

Ключевое поведение:
- без подписки вкладка ведёт на Paywall;
- с подпиской экран доступен;
- результат зависит от выбранного настроения.

## 10. Критерий готовности документа Screens

Описание экранов считается завершённым, если реализация следует следующим правилам:
- в приложении есть ровно 3 основных экрана;
- каждый экран имеет чётко определённую роль;
- структура каждого экрана предсказуема и мобильна;
- табы ведут себя однозначно;
- locked/premium-flow не требует догадок;
- состояние после подписки не ломает навигацию;
- layout устойчив на мобильном экране;
- экраны не расширяются лишними flow и лишними сущностями.
