# Stack.md

## 1. Технологический стек

Проект реализуется на следующем стеке:
- **React Native**
- **Expo (managed workflow)**
- **TypeScript**
- **React Navigation**
- **Bottom Tabs Navigation**
- **AsyncStorage** для локального сохранения состояния подписки
- **react-native-safe-area-context** для корректной работы с safe area
- **expo-linear-gradient** для фонов, карточек и premium-акцентов
- **Expo Image** или стандартный `Image` для локальных ассетов
- **Animated API** или лёгкие встроенные анимации React Native для микроанимаций

Проект не должен требовать нативной ручной настройки вне обычного Expo managed workflow.

---

## 2. Причина выбора стека

Данный стек выбран по следующим причинам:
- он максимально подходит для быстрого создания мобильного прототипа;
- React Native + Expo даёт быстрый запуск и предсказуемую структуру проекта;
- TypeScript повышает управляемость и снижает риск хаотичного кода;
- React Navigation является стандартным и понятным решением для мобильной навигации;
- AsyncStorage позволяет сохранить пользовательское состояние без backend;
- safe area и адаптивная вёрстка закрывают ключевые требования тестового задания;
- стек достаточно простой, чтобы не перегружать проект лишней инфраструктурой.

---

## 3. Жёсткие ограничения по стеку

В рамках проекта необходимо соблюдать следующие ограничения:
- использовать только **React Native + Expo**;
- не переходить на Flutter;
- не использовать Expo bare workflow;
- не добавлять backend;
- не добавлять real payment SDK;
- не добавлять real AI SDK;
- не подключать сложные state-management библиотеки, если локального состояния достаточно;
- не усложнять архитектуру ради будущего масштабирования.

Проект должен оставаться компактным и полностью соответствовать MVP-прототипу.

---

## 4. Общая архитектурная модель

Архитектура проекта должна быть простой и предсказуемой.

Принципы:
- логика разделена по ответственности;
- UI-компоненты переиспользуемы;
- данные медитаций и мок-ответов вынесены отдельно;
- навигация отделена от экранов;
- состояние подписки не размазывается хаотично по приложению;
- локальное хранилище изолировано в отдельном модуле;
- проект не должен быть монолитом из одного большого файла.

## 6. Навигация

Навигация реализуется через **React Navigation**.

Требования:
- использовать **bottom tabs** как основной способ переключения между разделами;
- табы: **Медитации**, **Настрой дня**, **Premium**;
- стартовый экран приложения: **Медитации**;
- переходы на Paywall из locked-состояний должны быть предсказуемыми и явными;
- не использовать сложные вложенные навигационные схемы без необходимости.

Дополнительно:
- стандартный header React Navigation не является обязательным;
- заголовки экранов реализуются как часть кастомного UI внутри экранов.

---

## 7. Управление состоянием

Для текущего проекта достаточно простого локального управления состоянием.

Необходимые состояния:
- статус подписки;
- состояние загрузки подписки из хранилища;
- выбранное настроение на AI Mood экране;
- состояние генерации текста;
- сгенерированный мок-результат.

Предпочтительный подход:
- `useState`
- `useEffect`
- при необходимости `Context`, если это действительно упрощает передачу `isSubscribed`

Не использовать тяжёлые решения типа Redux, MobX, Zustand, если без них можно обойтись.

---

## 8. Локальное хранение данных

Для сохранения подписки используется **AsyncStorage**.

Что сохраняется:
- только факт активации подписки;
- сохранение должно переживать перезапуск приложения.

Что не сохраняется обязательно:
- временное выбранное настроение;
- временный сгенерированный текст;
- внутренние анимационные состояния.

Хранилище должно быть изолировано в отдельном модуле, а не вызываться напрямую в произвольных местах приложения.

---

## 9. Работа с данными

В проекте используются только локальные демонстрационные данные.

Источники данных:
- локальный массив карточек медитаций;
- локальный массив тарифов подписки;
- локальный набор мок-ответов для AI Mood;
- локальные изображения ассетов.

Запрещается:
- делать обязательные запросы к серверу;
- строить архитектуру под API, которого в MVP нет;
- внедрять лишний network-layer.

---

## 10. Работа с AI-функцией

AI Mood реализуется без реального AI API.

Технический подход:
- логика генерации вынесена в отдельный сервис или helper;
- сервис принимает выбранное настроение;
- на выходе возвращает один из заранее заготовленных реалистичных текстов;
- допускается небольшая искусственная задержка для имитации генерации.

Это поведение должно выглядеть как AI-фича, но оставаться полностью локальным и контролируемым.

---

## 11. Работа с изображениями и ассетами

В проекте используются:
- один общий фон приложения;
- отдельные локальные изображения для карточек медитаций.

Требования:
- изображения должны храниться локально внутри проекта;
- пути к изображениям должны быть предсказуемыми и не размазываться по коду;
- карточки должны использовать изображения аккуратно, без ломания layout;
- фон не должен ухудшать читаемость контента.

---

## 12. Styling-подход

Стиль должен быть организованно и системно описан в коде.

Предпочтительный подход:
- использовать `StyleSheet.create`;
- вынести токены в `constants`;
- централизовать цвета, отступы, радиусы, типографику и тени;
- не дублировать магические числа по разным файлам без необходимости.

Допускается использование:
- `LinearGradient` для премиального визуала;
- мягких overlay-слоёв;
- полупрозрачных блоков;
- аккуратных теней и скруглений.

Не использовать:
- хаотичный inline styling в большом объёме;
- абсолютное позиционирование как основу layout;
- жёстко прибитые размеры без необходимости.

---

## 13. Обязательные mobile-требования к layout

### 13.1. Safe Area
Каждый экран обязан учитывать safe area.

Требования:
- контент не должен заходить под статус-бар;
- верхние заголовки и ключевые блоки должны располагаться внутри безопасной зоны;
- нижние элементы не должны конфликтовать с таб-баром и нижней системной зоной.

### 13.2. Scrollable layout
Ключевые экраны должны корректно прокручиваться, если контент не помещается.

Особенно важно:
- Paywall не должен предполагать фиксированную высоту экрана;
- длинные блоки контента должны жить внутри `ScrollView` или аналогичного безопасного контейнера.

### 13.3. Без критичных absolute-решений
Не использовать `position: absolute` для основной структуры экранов.

Допускается только:
- мелкий декоративный слой;
- не-критичный визуальный акцент;
- overlay, не влияющий на доступность основного контента.

### 13.4. Без ненужных фиксированных высот
Не задавать жёсткие высоты блокам, если это не обязательно по дизайну.

Причина:
- фиксированные размеры чаще всего ломаются на маленьких экранах;
- текст и карточки должны иметь возможность естественно расти по контенту.

---

## 14. Touch targets и интерактивность

Все интерактивные элементы должны быть удобны для нажатия пальцем.

Требования:
- кнопки и ключевые элементы управления — не менее 44–48 px по высоте;
- карточки должны иметь понятную зону нажатия;
- выбранные состояния должны быть визуально различимыми;
- нажатие должно иметь заметный pressed feedback.

Особенно это касается:
- карточек медитаций;
- тарифных карточек на Paywall;
- кнопки активации подписки;
- выбора настроения на AI Mood.

---

## 15. Анимации

Анимации должны быть лёгкими и безопасными.

Разрешённые сценарии:
- мягкий scale или opacity feedback на нажатие;
- плавное визуальное выделение выбранного тарифа;
- мягкое выделение выбранного настроения;
- деликатное появление AI-результата.

Не делать:
- тяжёлые сложные анимационные последовательности;
- анимации, влияющие на стабильность layout;
- избыточные декоративные эффекты.

---

## 16. Производительность и простота

Проект небольшой, поэтому приоритет — простота и стабильность.

Необходимо:
- избегать лишних абстракций;
- не добавлять оптимизации, которые не дают пользы в MVP;
- не усложнять код под будущую production-архитектуру;
- поддерживать быструю навигацию и отзывчивый UI.

---

## 17. Code style requirements

Требования к коду:
- TypeScript обязателен;
- компоненты должны быть небольшими и понятными;
- логика и отображение по возможности разделены;
- имена файлов и сущностей должны быть предсказуемыми;
- избегать дублирования;
- не создавать чрезмерную вложенность;
- не смешивать данные, storage-логику и UI в одном месте без причины.

---

## 18. Что не использовать без прямой необходимости

Не использовать без явной причины:
- Redux
- MobX
- Zustand
- react-query
- axios
- form-библиотеки
- сложные animation-библиотеки
- библиотеки для real billing
- библиотеки для real AI
- сторонние UI kits, которые создают избыточную сложность

Если стандартных средств React Native и Expo достаточно, нужно использовать именно их.

---

## 19. Технический итог

Техническая реализация должна быть:
- мобильной, а не вебоподобной;
- компактной;
- предсказуемой;
- безопасной для маленьких экранов;
- удобной для последующей доработки;
- полностью достаточной для реализации MVP ZenPulse.
